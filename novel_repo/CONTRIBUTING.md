# Contributing to novel-master

感谢你为 novel-master 提交改动。

这个项目不是“提示词仓库”，而是一套面向长篇小说的结构化创作协议。提交前请先理解这一点：我们优先接受能增强可执行性、可验证性和长期一致性的改动，不接受把项目变成泛泛写作建议合集的改动。

## 贡献目标

欢迎的改动：

- 补充或修正长篇创作的工作流
- 增强角色、世界观、卷纲、章节、伏笔之间的同步机制
- 提高规则的可执行性与可检查性
- 增加脱敏样例、用法示例、回归测试样例
- 修复明显冲突、遗漏或自相矛盾的规范

谨慎接受的改动：

- 大幅调整默认字数、章节结构、评估标准
- 改变角色档案或世界观模板的核心字段
- 改动 CLI 行为或默认项目目录结构

原则上不接受：

- 写成空泛经验谈的“创作建议”
- 把具体作品私货、专属设定、专属术语直接写入 skill
- 只适配单一题材、单一作者习惯的规则
- 无法验证、无法执行、无法落地到正文的约束

## 提交前检查

请至少确认以下几点：

1. 改动是否服务长篇创作，而不是短篇随手写作
2. 改动是否可执行，而不是抽象口号
3. 改动是否与现有工作区优先原则冲突
4. 改动是否会破坏角色、世界、卷纲、章节之间的同步关系
5. 改动中是否混入了具体作品名称、角色名、私有术语或未脱敏示例

## 文档改动要求

- README 应面向首次使用者，优先回答“这是什么、怎么用、适合谁”
- SKILL.md 应面向执行规则，优先保证完整、精确、可落地
- CHANGELOG 应记录对行为和输出预期有影响的变更
- 示例文档必须脱敏，不得引用私有项目正文

## 规则改动要求

新增规则时，请尽量满足以下标准：

- 能指出适用场景
- 能说明为什么要有这条规则
- 能被检查，而不只是“感觉更好”
- 能落到角色档案、世界观、卷纲、章节或质检流程中的至少一个环节

优先使用这类表达：

- “必须包含……”
- “至少检查……”
- “若发生……应同步……”
- “禁止……”

尽量避免这类表达：

- “建议写得更有感觉”
- “尽量自然”
- “最好不要太 AI”

## 模板改动要求

修改 references/ 或 assets/ 中的模板时，请同步说明：

- 这项改动影响哪类用户
- 是新增字段、删减字段，还是字段重组
- 会不会影响旧项目兼容性
- 是否需要同步更新 README、CHANGELOG 或 CLI

## CLI 改动要求

scripts/novel-cli.js 当前定位是轻量脚手架，而不是完整生成系统。

如果你修改 CLI，请避免：

- 引入不可控的外部依赖
- 把 CLI 设计成强绑定单一目录结构的私有工具
- 在未说明的情况下改变已有命令语义

如果改动影响输出模板，请同时更新文档和样例。

## Issue 建议格式

提交问题时，优先包含：

- 使用场景
- 当前行为
- 预期行为
- 受影响的文件
- 是否涉及兼容性变化

## Pull Request 建议格式

- 改动目标
- 改动范围
- 为什么需要这项改动
- 是否影响现有输出协议
- 是否同步更新 README / CHANGELOG / 示例

## 许可证与生成内容

请不要默认把仓库许可证直接套用到用户用本 skill 生成的小说正文。

仓库本身的代码、模板、文档是否开源，与用户使用它生成的作品版权，是两个问题。提交文档或许可证相关改动时，请保持这一区分。
